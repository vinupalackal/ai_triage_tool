CODE LOGS DOCS

Teaching an AI to find bugs

A plain-English walkthrough of how to store source code, device logs, and documents so an AI can connect them — and point at the actual line of code that's broken.

No prior AI background needed. Every term is explained the first time it shows up.

Before we start

Why can't you just paste everything into ChatGPT?

Imagine a set-top box crashes in a customer's living room. To figure out why, an engineer needs three completely different kinds of information:

📄 the code

Millions of lines of C/C++ across dozens of repositories.

📊 the logs

Millions of log lines a day, from thousands of devices.

📚 the docs

Design specs, wikis, and a decade of old bug reports.

Each one is shaped differently, is searched differently, and grows at a different speed. Logs are millions of near-identical lines that need to be counted and filtered. Code needs to be understood by structure — which function calls which — not just by the words in it. Docs need to be searched by meaning — "why does the tuner drop signal" should find a page that never uses the word "tuner."

Analogy It's like asking one person to be your librarian, your GPS, and your accountant at the same time. Each job needs a different tool. The trick isn't finding one tool that does everything — it's building three good tools and teaching an AI how to use all three together.
Vocabulary first

Six words you'll need

These terms show up constantly in AI system design. Here's what they actually mean, with no jargon.

LLM (Large Language Model)

The "brain" — e.g., Claude or GPT. It's very good at reading text and reasoning about it, but it doesn't automatically know your codebase or your logs. You have to bring the right information to it.

Embedding

A way of turning a piece of text (or code) into a list of numbers that captures its meaning. Two pieces of text about similar things end up with similar numbers, even if they don't share any words.

Vector database

A database built to store those lists of numbers and quickly answer "which other stored items have numbers closest to this one?" — i.e., "find me the most similar things."

RAG (Retrieval-Augmented Generation)

Instead of asking the AI to answer from memory, you first retrieve the relevant documents/logs/code, hand them to the AI as context, and then ask it to answer. This is how you stop the AI from guessing.

Knowledge graph

A map of how things connect — "this commit touched this file," "this file belongs to this component," "this component logged this error." A graph answers connection questions; a vector database answers similarity questions.

Agent (tool-calling)

An AI that doesn't just answer — it can take actions, like "search the logs," "look up this function," "run this query" — look at the result, and decide what to do next. This is what lets the AI investigate a bug instead of just guessing from one snapshot of context.

The big picture

Three streams, one investigator

Here's the shape of the whole system before we zoom into each piece. Code, logs, and documents each get stored in the format that suits them best. An AI "agent" sits on top and pulls from all three when it investigates an issue.

Each data type gets its own storage format — a shared ID system and a knowledge graph tie them together. An AI agent (next section) queries all three.
Data type 1 of 3

Device Logs

A single set-top box can produce thousands of log lines a day. Across a fleet of a million devices, that's an unimaginable volume of near-identical text. Storing it as raw text and searching it like a Word document simply doesn't scale.

Analogy Imagine a security guard writing down every single time a door opens: "Door 4 opened at 9:01:03am. Door 4 opened at 9:01:47am. Door 7 opened at 9:02:01am." After a million entries, you don't want to read them — you want to know "how many times did Door 4 open today, and were any of those unusual?"

Step 1 — Turn messy text into templates

The first trick is a technique called log templating (a popular tool for this is called Drain3). It looks at thousands of log lines and notices that most of them are the same sentence with different numbers plugged in:

Millions of raw lines collapse into a small, countable list of "log signatures."

Now instead of a mountain of text, you have a short list of named events with counts — something both a computer and a human can reason about ("this signature spiked 40x right after the last firmware update").

Step 2 — Store it in a format built for counting, not for essays

Logs get stored in what's called a columnar database (an example is ClickHouse) — think of it as a spreadsheet engine built to scan billions of rows in under a second, rather than a search engine built to find one needle in a haystack. This matches how logs are actually used: "count how often X happened, grouped by device model, over the last week."

OptionWhat it's good atWhy it fits (or doesn't)
Raw text filesNothing, at scaleNo structure, slow, expensive to search
Search engine (e.g. Elasticsearch)Finding specific words in textGood for "find the needle," worse at "count everything"
Columnar database (e.g. ClickHouse)Aggregating and counting huge volumes fastBest fit for logs/telemetry at fleet scale

What if you only have one device, not a million?

Everything above is built for fleet scale — but most investigations don't start at fleet scale. They start with one engineer, one crashing device, and a log file they just pulled over SSH. Standing up a streaming pipeline and a columnar database for that would be massive overkill.

The fix is to design the templating step as one portable tool with two modes, instead of building a separate "small" tool and a separate "big" tool:

The same engine that runs continuously across a fleet can be pointed at a single log file for a one-off investigation — with no infrastructure and no format change.
Analogy Think of a recipe that scales — the same recipe works for a dinner for two or a restaurant kitchen feeding 500 people a night. You don't need a different recipe, just a bigger kitchen. The templating engine works the same way: same "recipe" for turning logs into patterns, whether it's reading one file on a laptop or a live stream from a million devices.

What to do instead of fleet statistics, when you only have a few devices

With a fleet, you can ask "did this event spike after the last update?" With one or two devices, there's no fleet to compare against — so the useful trick becomes diffing instead of counting: run the same templating tool against a failing device's log and a healthy device's log, then compare the two short lists of templates instead of reading either log in full.

Diffing two short template lists finds the anomaly directly — no fleet-wide statistics required.

The key design promise is that this scales up cleanly: the same signature IDs produced during a small, ad-hoc investigation are the exact same IDs the fleet system would produce for the same event later — so nothing done at small scale is ever thrown away once a device population grows into a full fleet.

Data type 2 of 3

Source Code

Code is different from prose. What matters isn't just "what words are in this file" — it's structure: which function calls which, which file defines a symbol, and which commit last touched a given line.

Analogy Searching code by keyword similarity is like trying to find a specific house by describing what its front door looks like. Searching code by structure is like using its actual street address. For debugging, you need the address — the exact function, the exact commit, the exact line — not just "something that looks similar."

Two complementary techniques

  • Embeddings for code (semantic search): lets you ask "find code that handles HDMI reconnection" in plain English, even if you don't know the function name. Good for exploring.
  • Precise code indexing (an example is a format/tool called SCIP, from Sourcegraph): this is compiler-accurate — it knows, with certainty, that function A calls function B, because it actually parsed the code the way a compiler would. Good for pinpointing.
Precise code indexing turns a crash symbol into an exact file, line, and commit — then the code graph shows what's connected to it.
Data type 3 of 3

Documents & Past Bug Reports

This is the category people usually mean when they say "RAG." Design specs, wiki pages, and years of old bug tickets are unstructured text — the goal is to find the handful of paragraphs relevant to the issue at hand, out of tens of thousands of pages.

How embeddings actually work, step by step

"HDMI drops after standby" and "HDMI signal lost after sleep" share almost no words — but an embedding still recognizes them as the same problem.
Important nuance Embeddings are great at meaning, but surprisingly bad at exact things — an error code like ERR_0x8004, a parameter name, or a function name. For those, good old-fashioned keyword search (the technology behind most search engines, called BM25) actually wins. That's why real systems use both together — this is called hybrid search, covered next.
Bringing search together

Hybrid search: don't pick one

A common mistake is assuming "AI search" means "vector search only." In practice, the best results come from running two (or three) searches at once and combining the results:

Three different search strategies run in parallel, then their results are merged into a single ranked list.
Why this matters here A device error code, a TR-181 parameter name, or a function symbol needs an exact match. A vague description of symptoms needs meaning-based matching. A question like "what else in the codebase depends on this driver" needs the graph. Real triage questions are usually a mix of all three — so the search layer has to be too.
The connective tissue

The knowledge graph: how everything gets linked

Vector search and keyword search both answer "what looks similar to my query?" Neither one, on its own, can answer a question like "which recent commits touched the component that this crash came from?" That's a relationship question, and relationship questions are what a knowledge graph is for.

Think of it as a map with dots (called nodes) and lines connecting them (called edges). Each dot is a real thing — a commit, a file, a component, a type of log message, a past incident. Each line is a real relationship — "belongs to," "touched," "caused," "mentions."

A tiny slice of a knowledge graph. Follow the lines and you can answer: "this crash's log signature belongs to the Tuner component, which this commit modified, which is documented here, and has caused two prior incidents."
Analogy A vector database is like asking "what does this remind you of?" A knowledge graph is like asking "what is this actually connected to?" You need both — one for fuzzy recall, one for hard facts.
The unglamorous but critical part

What actually ties the three stores together

None of this works unless every piece of data — a log line, a crash report, a commit, a document — is tagged with the same set of shared labels. This is the single most important design decision in the whole system, and it's the easiest one to skip.

Shared labelExample valueWhy it matters
firmware_build_idRDKV-4.2.118A crash can only be decoded using the debug symbols from the exact build that produced it
componentTunerRoutes an issue to the right owning team and links logs to code and docs about the same subsystem
commit_shaa91f3c2Connects a code change to the incidents and log signatures that appeared after it shipped
log_signature_idsig_00417The stable "fingerprint" for a templated log message, from Step 1 of the logs section
device_modelXR400Lets you see whether an issue is fleet-wide or limited to one hardware family

If two systems disagree on what to call the same firmware build, or the same component, the AI can't connect them — no matter how good the search is underneath.

Now add the investigator

The AI agent: a detective, not a search bar

Storing the data well is half the job. The other half is how the AI actually uses it. The pattern that works well is called an agent loop (sometimes called ReAct, short for "reason and act"): instead of answering in one shot, the AI is given a set of tools and allowed to investigate step by step, the way a person would.

The loop repeats — often 5 to 15 times — until the agent has enough evidence to state a root cause with confidence.

The "tools" in step 2 are exactly the storage systems described above, wrapped so the AI can call them:

search_logs()

Query the log store for a signature, time range, or device model.

symbolicate_crash()

Turn a raw crash address into a function name, file, and line.

search_code()

Find relevant functions by meaning or by exact symbol.

graph_query()

Ask the knowledge graph a connection question.

search_docs()

Find relevant specs or past incident write-ups.

blame_history()

See which commits touched a file or function recently.

Seeing it all work together

A worked example: chasing a real crash

Say a set-top box, model XR400, running firmware RDKV-4.2.118, keeps rebooting a few minutes after standby. Here's what the agent loop above might actually do, end to end:

#StepSourceWhat happens
1Crash comes inlogsThe device auto-uploads a crash dump, tagged with the firmware build ID and device model — the shared labels from Section 8.
2SymbolicatecodeThe crash address resolves to tuner_lock_retry() at drivers/tuner/lock.c:142, using symbols that match build RDKV-4.2.118 exactly.
3Check historylogsThis crash's log signature has spiked 40x in the last 3 days — right after the last firmware rollout.
4Walk the code graphcode"What commits touched lock.c recently?" surfaces commit a91f3c2 — "adjust retry backoff timing."
5Pull related docsdocsA hybrid search for "tuner retry standby" finds the tuner design spec and a two-year-old incident with the same symptom, even though it was worded as "reboot loop," not "crash."
6Propose root causeall threeThe agent states a root cause and links every claim back to its source.
Final answer from the agent "This crash is very likely caused by commit a91f3c2, which changed retry timing in tuner_lock_retry(). It correlates with the timing of the firmware rollout, matches a log signature that spiked 40x, and resembles incident #3877 from two years ago. Recommend reviewing the backoff logic against the tuner spec." Each claim links back to its source, so an engineer can verify it in seconds instead of minutes.
Getting practical

Do we build this, or buy it?

You don't need to build all of this from scratch. A sensible split:

LayerPractical approach
Log storage & searchBuy / adopt an existing platform (there are strong open-source and commercial options) rather than build one
Code indexingAdopt an existing code-intelligence tool rather than write your own parser for every language
Crash symbolicationAdopt an existing crash-reporting/symbolication service
The glue: shared IDs, the knowledge graph, the agent's toolsBuild this yourself — it's specific to your codebase, your device fleet, and your data, and no off-the-shelf product understands your embedded platform's quirks out of the box

The unglamorous "glue" layer is where almost all of the real value lives, precisely because it's the one part nobody else can sell you pre-made.
